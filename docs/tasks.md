- The engine now performs a bounded reconnect with backoff when an outbound authenticated transport send hits an I/O failure, and a scripted host runtime test verifies that it re-handshakes and retries the payload successfully.
- The engine now also routes receive-loop transport failures and periodic keepalive send failures through the same bounded reconnect path, with scripted host runtime tests covering both recovery triggers.
- A separate Switch integration app now stages as its own NRO and exercises the real `swg::Client` plus `swg::AppSession` flow on hardware without coupling test scenarios to the manager UI.

- The SDK now also exposes a route-aware `SessionSocket` wrapper that composes `GetNetworkPlan`, `ResolveDns`, and session packet I/O into one Moonlight-facing transport surface for direct, tunnel-packet, and deny cases.

- The Switch integration app now drives its DNS and socket-helper diagnostics from the active profile `endpoint_host`, and it skips DNS probes cleanly when the profile uses a numeric endpoint instead of a hostname.
- The SDK and control service now expose a real UDP tunnel-datagram handle API (`OpenTunnelDatagram` / `SendTunnelDatagram` / `RecvTunnelDatagram` / `CloseTunnelDatagram`) plus `swg::TunnelDatagramSocket`, with host regression coverage for a Moonlight-style video datagram round trip.
- The SDK and control service now expose a real TCP tunnel-stream handle API (`OpenTunnelStream` / `SendTunnelStream` / `RecvTunnelStream` / `CloseTunnelStream`) plus `swg::TunnelStreamSocket`, with host regression coverage for a Moonlight-style HTTPS control round trip.
- The WireGuard engine's authenticated receive queue remains bounded but is now sized for media bursts (`256` packets / `512 KiB` instead of `8` packets / `8 KiB`), and host regression coverage now queues a video-like inbound burst before the client drains it.
- Tunnel datagram diagnostics now log sparse `audio datagram` and `video datagram` receive/pending/close events plus sparse UDP tuple-mismatch warnings, so on-device Moonlight runs can distinguish `no 47998/48000 media reached SWG` from `media arrived on the wrong source/destination port tuple`.
- Recent on-device logs now show the audio handle does receive matched payloads while video still stays at `recv_bytes=0`. The sysmodule now reserves Moonlight video traffic on local UDP port `50001` to match the RTSP-advertised client port pair `50000-50001` instead of using a synthetic `20001` port.
- The `50001` video-port change is now confirmed in live logs, but it did not restore video. The next root-cause hypothesis proved more specific: the inbound sysmodule path was matching only fully reassembled inner IPv4 packets, so large fragmented video UDP could arrive invisibly while audio still worked. The sysmodule now performs bounded inner-IPv4 fragment reassembly before datagram/stream matching, with a host regression covering fragmented Moonlight-style video UDP.
- Newer Moonlight repro logs then exposed a second fragment-path bottleneck during sustained playback: repeated `dropping fragmented IPv4 reassembly reason=fragment entry limit` warnings while media was active. The latest logs proved the `128`-entry build was still saturating on full-size one-fragment orphan assemblies (`active_assemblies=128`, `fragment_storage_bytes=362496`), so the sysmodule now raises concurrent IPv4 fragment assemblies to `256`, raises bounded fragment storage to `768 KiB`, and host regression coverage now includes a 192-packet Moonlight-style fragmented video burst with `1408`-byte payloads and tail fragments arriving before head fragments.
- Fresh on-device profiling then showed a second user-visible bottleneck in the client, not just the sysmodule: Moonlight's main core pegs at `100%` once video starts. The current local hypothesis matched the Switch call site in `SwgPlatformSocketsCompat.cpp`: zero-timeout poll loops were turning every empty tunnel wait into a real compat-bridge `Receive()` IPC probe on the caller thread. `sdk/src/compat_bridge_switch.cpp` now rate-limits empty zero-timeout stream/datagram receive probes with a small monotonic backoff so repeated `poll(..., timeout=0)` slices stop hammering `ReceiveTunnelDatagram` / `ReceiveTunnelStream` when no packet is ready.
- The direct-vs-SWG comparison then falsified decode/render as the root cause and narrowed the hot path further: tunneled video was still paying one full `RecvTunnelDatagram` control request per UDP packet. The control plane now exposes a bounded `RecvTunnelDatagramBurst` IPC, `sdk/src/compat_bridge_switch.cpp` now queues burst results locally instead of fetching one datagram per Moonlight recv call, host `swg_tests` covers a 4-packet burst receive, and both the staged sysmodule package and `Moonlight.nro` rebuild cleanly.
- A follow-up hardware repro still showed very high CPU even on a mostly static image, which narrowed the next local bottleneck from packet count to idle waiting behavior: Moonlight's UDP receive threads use a `100 ms` timeout, and the compat bridge was still burning that timeout in `10 ms` app-side `ReceiveBurst` probes. `TunnelDatagramBurstRequest` now carries `timeout_ms`, `sysmodule/src/local_service.cpp` now performs the bounded wait inside the sysmodule, and `sdk/src/compat_bridge_switch.cpp` now issues one timed burst request for blocking UDP reads instead of spinning ten times per timeout window from the Moonlight process.
- That receive-side change improved CPU and stabilized audio, but the next hardware repro exposed a separate control-path symptom: input was delayed by `1-2` seconds even without Moonlight warnings. The Moonlight-side ENet flush hypothesis was falsified on hardware and reverted. The new SWG-only hypothesis is shared control-socket head-of-line blocking: the compat bridge's tunneled control-datagram wait could monopolize the shared `swg:ctl` transport long enough to delay input sends on the same ENet control socket. `sdk/src/compat_bridge_switch.cpp` now keeps `StreamControl` datagram waits non-blocking and sleeps outside the per-socket lock so input sends can interleave with control receive polling again.
- Video/audio/control now work end-to-end on hardware, so the current bottleneck has shifted to quality under load. The sysmodule now routes packets that already match an open datagram or stream directly into per-flow pending queues instead of repeatedly reparsing them through the shared `deferred_packets_` backlog on every poll. This is intended to cut packet latency and queue growth during Moonlight streaming.
- The sibling Moonlight-Switch Switch build now embeds `swg_common` and `swg_sdk`, and the Moonlight-oriented tunnel bridge implementation now lives in `swg_sdk` behind reusable C/C++ headers instead of a large app-local translation unit.
- The sibling Moonlight-Switch Switch build now keeps only thin wrapper files for HTTP response adaptation plus a forwarding bridge header, while recognized TCP, UDP, and ENet transport sockets still route through the SDK-owned bridge onto `TunnelStreamSocket` and `TunnelDatagramSocket`.

- The SDK bridge now exposes neutral `swg/compat_bridge*.h` headers above the existing implementation, Moonlight headers are backward-compatible wrappers, and the Switch integration app includes a non-Moonlight compat resolve probe.
- A manual host live-handshake probe now loads a real config file and runs the same local-control-service `Connect()` path against a live endpoint for off-device handshake validation.
